Don't comment the obvious code

Posted on 2020-02-05 | In tips

In fact, you should write as little comments as possible.

Basics

Comments are great tool and are very helpful when you learn to code or code you are working on is hard to read, but with time and practice you will read and understand code faster than you could read code comments.

Also:

  • by not writing comments you will learn to write self commenting code. Lets try this example:

  • Comments can lie, code not.
    See my favourite: 

    /** Adds two integers */
public int Calculate(int i1, int i2)
{
  return i1 - i2;
}

    Of course this one is easy, but imagine more complicated algorithm, when instead of reading actual code you read the comment and assume code is doing what comment says. That happens often when first programmer writes the method and the comment, then second one changes the method, but leaves the comment untouched. That leads us to:

  • Comments needs to be maintained, just as the code is.
    More code leads to more little things to care about when maintaing the system. That is also true for comments. The less code and comments you must write the easier maintenance of the system is.

So should you use comments at all?

Yes.

There are few things that you should explain in your code.

  • The hard to understand (low-level? heavy optimized?) code blocks.
  • Things that looks like errors but are intentional.
  • Methods that do more than they name suggests.

Also, when commenting your methods declaration use the style that are recognized by your IDE.
For C# it should be /// ..., but i prefer the javadoc comment blocks: /** ... */ (and most IDE’s recognize that one as well).
